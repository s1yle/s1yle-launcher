# WeCraft! Launcher - 文档维护规范

> **版本**: 1.0  
> **生效日期**: 2026-05-18  
> **适用范围**: 所有项目文档（AGENTS.md + docs/*.md）

---

## 一、文档分层架构

### 1.1 三层文档体系

```
Level 1: AGENTS.md (快速参考)
    ├─ 定位：开发者快速查阅的备忘录
    ├─ 规模：控制在 800 行以内
    ├─ 更新频率：高（每次重构后更新）
    ─ 内容：核心概念、常用命令、关键路径

Level 2: docs/*.md (详细文档)
    ├─ 定位：系统性学习和深度参考
    ├─ 规模：每个文件 300-600 行
    ├─ 更新频率：中（重大功能更新时）
    └─ 内容：架构设计、组件 API、使用示例

Level 3: 代码注释 (JSDoc/Rust doc)
    ├─ 定位：即时更新的开发文档
    ├─ 规模：与代码同步
    ├─ 更新频率：每次代码提交
    └─ 内容：函数说明、参数解释、返回值
```

### 1.2 文档职责边界

| 文档层级 | 应该包含 | 不应该包含 |
|---------|---------|-----------|
| **AGENTS.md** | 核心路由、状态管理表、配置示例、注意事项 | 详细实现代码、历史变更记录、完整 API 列表 |
| **docs/architecture.md** | 架构图、目录结构、配置系统详解 | 具体组件用法、API 调用示例 |
| **docs/components.md** | 组件 Props 接口、使用示例 | 组件内部实现细节 |
| **docs/api.md** | API 调用示例、错误处理 | Rust 后端实现细节 |
| **docs/changelog.md** | 重大重构、新功能、Breaking Changes | 日常 bug 修复、代码优化 |

## 二、文档更新触发条件

### 2.1 必须更新文档的场景

#### 🟢 高优先级（立即更新）

1. **新增/删除路由**
   ```markdown
   触发条件：router/config.tsx 变更
   更新文件：AGENTS.md §4 + docs/architecture.md §5
   更新内容：路由表、已移除页面列表
   ```

2. **新增/删除 Zustand Store**
   ```markdown
   触发条件：src/stores/ 目录变更
   更新文件：AGENTS.md §5 + docs/architecture.md §4
   更新内容：Store 表格、使用示例
   ```

3. **配置系统变更**
   ```markdown
   触发条件：config/index.ts 或 config/types.ts 变更
   更新文件：AGENTS.md §6 + docs/architecture.md §3
   更新内容：配置项列表、使用示例
   ```

4. **新增通用组件**
   ```markdown
   触发条件：src/components/common/ 新增组件
   更新文件：AGENTS.md §8 + docs/components.md
   更新内容：组件列表、Props 接口、使用示例
   ```

5. **后端 API 变更**
   ```markdown
   触发条件：src-tauri/src/commands/ 或 src/helper/rustInvoke.ts 变更
   更新文件：AGENTS.md §7 + docs/api.md
   更新内容：API 命令表、调用示例
   ```

####  中优先级（本周内更新）

1. **UI 架构重大调整**
   ```markdown
   触发条件：布局组件重构（如灵动岛系统）
   更新文件：AGENTS.md §9 + docs/architecture.md §6
   更新内容：布局结构图、切换逻辑
   ```

2. **编码规范更新**
   ```markdown
   触发条件：代码审查中发现的共性问题
   更新文件：AGENTS.md §10
   更新内容：命名规范、最佳实践
   ```

3. **注意事项新增**
   ```markdown
   触发条件：遇到难以排查的 Bug 或踩坑
   更新文件：AGENTS.md §12
   更新内容：避坑指南、解决方案
   ```

#### 🔵 低优先级（月度整理）

1. **技术栈版本升级**
   ```markdown
   触发条件：package.json 或 Cargo.toml 重大版本更新
   更新文件：AGENTS.md §2
   更新内容：技术栈列表
   ```

2. **目录结构调整**
   ```markdown
   触发条件：src/ 或 src-tauri/src/ 目录重构
   更新文件：AGENTS.md §3 + docs/architecture.md §2
   更新内容：目录树
   ```

#### ⭐ AI 全权实现功能

**触发条件**：AI 完成某个功能的完整实现后

**文档更新流程**：
```markdown
AI 实现完成
    ↓
AI 输出文档更新建议清单
    ↓
开发者审查代码 + 测试功能
    ↓
开发者根据清单更新文档
    ├─ Level 1: AGENTS.md（新增路由/Store/组件）
    ├─ Level 2: docs/*.md（详细实现说明）
    └─ Level 3: 代码注释（JSDoc/Rust doc）
    ↓
在 changelog.md 中记录
```

**AI 的责任**：
- 列出所有新增/修改的文件
- 指出需要更新文档的位置
- 提供文档更新建议内容
- 标注文档更新优先级

**开发者的责任**：
- 审查代码质量（类型定义、命名规范）
- 测试功能是否正常（路由、API、UI）
- 根据 AI 建议更新文档
- 在 changelog.md 中记录本次实现

**示例**：
```markdown
AI 实现玩家管理页面后，输出：

📋 文档更新建议

高优先级（必须更新）：
1. AGENTS.md §4 核心路由
   - 添加：/admin/players
2. AGENTS.md §5 状态管理
   - 添加：playerStore 到表格
3. docs/architecture.md §2 目录结构
   - 新增：src/pages/admin/Players.tsx

中优先级（建议更新）：
1. docs/components.md
   - 新增：PlayerList, PlayerItem 组件文档
2. docs/api.md
   - 新增：get_players, add_player 调用示例

低优先级（可选更新）：
1. docs/changelog.md
   - 记录本次功能实现
```

## 四、文档编写规范

### 4.1 文件路径引用

**核心原则**: 所有文档引用必须使用相对路径，禁止使用绝对路径

```markdown
✅ 正确：
- 同目录文件：[`architecture.md`](architecture.md)
- 子目录文件：[`docs/architecture.md`](docs/architecture.md)
- 父目录文件：[`../src/helper/rustInvoke.ts`](../src/helper/rustInvoke.ts)
- 源码文件：[`src/router/config.tsx`](src/router/config.tsx)

❌ 错误：
- 模糊路径：见 docs 文件夹
- 错误格式：[architecture](docs/architecture)
- 绝对路径：[`src/router/config.tsx`](src/router/config.tsx)
```

**路径规范**:
- 从 `AGENTS.md` 引用 `docs/` 文件：使用 `docs/filename.md`
- 从 `docs/*.md` 引用同级文件：使用 `filename.md`
- 从 `docs/*.md` 引用 `src/` 文件：使用 `../src/path/to/file.tsx`
- 永远不要使用 `file:///` 协议

### 4.2 代码块规范

```markdown
✅ 正确：
```typescript
import { config } from '@/config';
const theme = await config.get('theme.mode');
```

❌ 错误：
```
import { config } from '@/config';  // 无语言标签
const theme = await config.get('theme.mode');
```
```

### 4.3 表格规范

```markdown
✅ 正确：
| Store | 文件 | 用途 |
|-------|------|------|
| `userRoleStore` | `src/stores/userRoleStore.ts` | 用户角色 |

❌ 错误：
| Store | 文件 | 用途 |
|-------|------|------|
| userRoleStore | src/stores/userRoleStore.ts | 用户角色 |
（缺少代码标记 `）
```

### 4.4 图表规范

**ASCII 架构图**（推荐用于 AGENTS.md）：
```
┌─────────────────────────────────────────────┐
│  [最小化] [最大化] [关闭]   ← FloatingControls
│
│        ┌──────────────────┐
│        │  🏝️ 灵动岛导航    │
│        └──────────────────┘
└─────────────────────────────────────────────┘
```

**Mermaid 流程图**（推荐用于 docs/*.md）：
```mermaid
graph LR
  A[用户操作] --> B[Store 更新]
  B --> C[组件重渲染]
  C --> D[UI 更新]
```

## 六、文档状态管理

### 6.1 文档分类与状态标识

**所有方案文档（`docs/plans/` 和 `docs/archives/`）必须在文档头部包含状态标识**：

```markdown
# 文档标题

> **创建日期**: YYYY-MM-DD  
> **最后更新**: YYYY-MM-DD  
> **状态**: [状态值]  
> **分类**: [分类名称]  
> **实施者**: [姓名/团队] (可选)
```

**状态值定义**：

| 状态 | 标识 | 说明 | 存放位置 |
|------|------|------|----------|
| **已实施** | ✅ | 功能已实现，文档有效，正在使用 | `docs/plans/` |
| **进行中** | 🚧 | 功能正在开发中 | `docs/plans/` |
| **已完成 (已归档)** | 📝 | 功能已完成，文档具有历史参考价值 | `docs/archives/` |
| **未实施** | ❌ | 功能未实现或已放弃 | `docs/archives/` |
| **已废弃** | ⚰️ | 功能已被其他方案替代 | `docs/archives/` |
| **待审查** | 🔍 | 状态不确定，需要人工审查 | `docs/archives/pending-review/` |

### 6.2 文档有效性验证规则

**AI 在查阅或引用方案文档时，必须执行以下验证**：

#### ✅ 验证步骤

1. **检查状态标识**
   ```markdown
   - 仅引用状态为"已实施 ✅"的文档
   - 避免引用状态为"未实施 ❌"、"已废弃 ⚰️"的文档
   - 遇到"待审查 🔍"文档时，提示用户需要人工确认
   ```

2. **验证代码实现**
   ```markdown
   - 检查文档中提到的核心文件是否存在
   - 检查代码中是否有对应的实现
   - 检查路由、组件、Store 等是否已注册
   ```

3. **检查时效性**
   ```markdown
   - 如果文档超过 3 个月未更新，标记为"可能过时"
   - 如果文档描述的功能已被重构，更新状态为"已废弃 ⚰️"
   - 如果发现文档与实际代码不符，标记为"待审查 🔍"
   ```

#### 📋 验证清单

AI 在引用方案文档前，应检查：

```markdown
- [ ] 文档状态是否为"已实施 ✅"？
- [ ] 文档中提到的核心文件是否存在？
- [ ] 代码中是否有对应的实现？
- [ ] 路由配置中是否有相关路径？
- [ ] 是否有最近 3 个月内的更新？
- [ ] 是否与其他文档冲突？
```

**如果任何一项检查失败**：
- 不引用该文档
- 在回答中说明"该方案文档可能过时/未实施"
- 建议用户查阅 `docs/archives/` 中的归档文档

### 6.3 AI 查阅和更新文档的规避规则

#### 🚫 禁止引用的文档

AI **不应引用**以下文档作为实现依据：

1. **状态为"未实施 ❌"的文档**
   ```markdown
   示例：`docs/archives/features/delete-game-folder-feature.md`
   原因：功能未实现，仅供参考
   ```

2. **状态为"已废弃 ⚰️"的文档**
   ```markdown
   示例：`docs/archives/configuration/layered-config-storage-design.md`
   原因：已被其他方案替代
   ```

3. **标记为"待审查 🔍"的文档**
   ```markdown
   示例：`docs/archives/pending-review/game-sidebar-redesign.md`
   原因：状态不确定，需要人工确认
   ```

4. **`docs/archives/` 中的历史文档**
   ```markdown
   例外：仅在用户明确询问历史方案时可以引用
   原因：这些是历史记录，不是当前实现
   ```

#### ✅ 推荐引用的文档

AI **应优先引用**以下文档：

1. **核心架构文档**
   ```markdown
   - `docs/architecture.md`
   - `docs/components.md`
   - `docs/api.md`
   ```

2. **`docs/plans/` 中状态为"已实施 ✅"的文档**
   ```markdown
   示例：`docs/plans/configuration/unified-config-implementation.md`
   ```

3. **最新更新的文档**
   ```markdown
   优先引用最近 3 个月内更新的文档
   ```

### 6.4 文档审查和归档流程

#### 📋 定期审查流程

**审查周期**：每季度一次

**审查责任人**：文档维护负责人

**审查内容**：

1. **检查 `docs/plans/` 中的文档**
   ```markdown
   - 是否有文档描述的功能已被移除？
   - 是否有文档超过 6 个月未更新？
   - 是否有文档与实际代码不符？
   ```

2. **检查 `docs/archives/pending-review/` 中的文档**
   ```markdown
   - 确认状态（已实施/未实施/已废弃）
   - 移动到正确的归档目录
   - 更新状态标识
   ```

3. **更新索引文件**
   ```markdown
   - 更新 `docs/plans/README.md`
   - 更新 `docs/archives/README.md`
   - 在 `docs/changelog.md` 中记录变更
   ```

#### 📦 文档归档流程

**触发条件**：

1. 功能已被新方案替代
2. 功能已被移除
3. 文档描述的内容已过时
4. 定期审查中发现的问题文档

**归档步骤**：

```markdown
1. 在文档头部添加状态标识
   - 状态：已完成 (已归档) 📝 / 未实施 ❌ / 已废弃 ⚰️

2. 移动到 `docs/archives/` 对应分类目录

3. 在文档开头添加归档说明
   ```markdown
   > 📝 **归档说明**
   > 
   > 此文档已归档，描述的功能可能已过时或被替代。
   > 请查阅 [`docs/plans/`](../plans/) 获取最新方案。
   > 
   > **归档日期**: YYYY-MM-DD
   > **替代文档**: [如有，填写替代文档路径]
   ```

4. 更新索引文件
   - 从 `docs/plans/README.md` 中移除
   - 添加到 `docs/archives/README.md`

5. 在 `docs/changelog.md` 中记录
   ```markdown
   ## YYYY-MM-DD - 文档归档
   - 归档：`xxx.md` → `archives/xxx.md`
   - 原因：功能已废弃/被替代
   ```
```

#### 🔍 文档有效性检查清单

**AI 在回答涉及方案文档的问题时，应执行**：

```markdown
1. 检查文档路径
   - 如果在 `docs/archives/` 中 → 提示"这是历史文档"
   - 如果在 `docs/plans/` 中 → 继续检查

2. 读取文档头部状态
   - 如果是"已实施 ✅" → 可以引用
   - 如果是其他状态 → 提示用户并避免引用

3. 验证代码实现（如可能）
   - 检查核心文件是否存在
   - 检查路由是否已配置
   - 检查组件是否已注册

4. 在回答中说明
   - "根据 [文档名](路径)（状态：已实施 ✅）..."
   - "注意：[文档名] 描述的功能可能已过时，请查阅最新文档"
```

## 八、文档过时判断标准

### 8.1 如何判断文档是否过时

**AI 和开发者在查阅文档时，应通过以下方法判断文档是否仍然有效**：

#### 🔍 快速判断方法

1. **检查文档头部状态标识**
   ```markdown
   ✅ 已实施 → 有效
   🚧 进行中 → 可能部分有效
   📝 已完成 (已归档) → 历史参考，可能过时
   ❌ 未实施 → 无效
   ⚰️ 已废弃 → 无效，已被替代
   🔍 待审查 → 需要人工确认
   ```

2. **检查最后更新日期**
   ```markdown
   - < 3 个月 → 通常有效
   - 3-6 个月 → 需要验证
   - > 6 个月 → 可能过时
   - > 1 年 → 很可能过时
   ```

3. **检查文档路径**
   ```markdown
   - `docs/plans/` → 通常是活跃文档
   - `docs/archives/` → 历史文档，可能过时
   - `docs/archives/pending-review/` → 状态不确定
   ```

#### 🧪 验证方法

**方法 1: 检查核心文件是否存在**

```markdown
步骤：
1. 读取文档中提到的核心文件路径
   示例：文档提到 "创建 src/config/index.ts"

2. 检查文件是否存在
   - 存在 → 继续验证
   - 不存在 → 文档可能未实施

3. 检查文件内容是否匹配
   - 文档描述的类/函数是否存在
   - 导出的 API 是否一致
```

**方法 2: 检查代码引用**

```markdown
步骤：
1. 搜索文档中提到的关键类/函数/组件
   示例：搜索 "UnifiedConfigManager"

2. 在代码中查找引用
   - 有引用 → 功能正在使用
   - 无引用 → 可能未实施或已移除

3. 检查路由配置
   示例：文档提到新增页面 /game-settings
   检查：src/router/config.tsx 中是否有该路由
```

**方法 3: 检查功能一致性**

```markdown
步骤：
1. 文档描述的功能 vs 实际代码
   - 功能是否完整实现
   - API 是否匹配
   - 配置项是否存在

2. 文档描述的数据结构 vs 实际类型定义
   - 字段是否一致
   - 类型是否匹配

3. 文档描述的 UI vs 实际界面
   - 组件是否存在
   - Props 是否匹配
```

### 8.2 文档过时的典型迹象

#### 🚩 红色警报（肯定过时）

```markdown
1. 文档描述的功能已被明确移除
   迹象：代码中找不到相关实现
   处理：标记为"已废弃 ⚰️"，移入 archives/

2. 文档描述的方案已被新方案替代
   迹象：有新的实现方案文档
   处理：标记为"已废弃 ⚰️"，添加替代文档链接

3. 核心文件已被删除
   迹象：文档提到的文件不存在
   处理：标记为"已废弃 ⚰️"

4. API 已完全改变
   迹象：文档中的 API 调用方式与实际不符
   处理：标记为"已废弃 ⚰️"，更新为新 API 文档
```

#### ⚠️ 黄色警报（可能过时）

```markdown
1. 文档超过 6 个月未更新
   迹象：最后更新日期久远
   处理：标记为"待审查 🔍"，需要验证

2. 文档描述的部分功能已改变
   迹象：部分 API 或配置项已改变
   处理：更新文档或标记为"待审查 🔍"

3. 有新的版本文档
   迹象：同一功能有多个版本文档
   处理：保留最新版，其他标记为"已废弃 ⚰️"

4. 代码已重构但文档未更新
   迹象：文件结构或命名已改变
   处理：更新文档或标记为"待审查 🔍"
```

#### ✅ 绿色标志（仍然有效）

```markdown
1. 文档最近 3 个月内有更新
2. 代码中有对应的实现
3. 有实际的代码引用
4. 与其他文档一致
5. 状态标识为"已实施 ✅"
```

### 8.3 AI 规避无用文档的具体规则

#### 🚫 绝对不要引用的文档

```markdown
1. `docs/archives/` 中的文档（除非用户明确询问历史）
   原因：这些是历史记录，不是当前实现

2. 状态为"未实施 ❌"的文档
   原因：功能未实现

3. 状态为"已废弃 ⚰️"的文档
   原因：已被其他方案替代

4. 状态为"待审查 🔍"的文档
   原因：状态不确定

5. 超过 1 年未更新的文档
   原因：很可能已过时
```

#### ✅ 应该优先引用的文档

```markdown
1. 核心架构文档
   - `docs/architecture.md`
   - `docs/components.md`
   - `docs/api.md`

2. `docs/plans/` 中状态为"已实施 ✅"的文档

3. 最近 3 个月内有更新的文档

4. 被其他文档频繁引用的文档（说明是权威参考）
```

#### ⚠️ 谨慎引用的文档

```markdown
1. `docs/archives/` 中的"已完成 📝"文档
   使用方式：可以作为历史参考，但需说明

2. 3-6 个月未更新的文档
   使用方式：需要验证后引用

3. 描述部分功能的文档
   使用方式：可能不完整，需结合其他文档
```

### 8.4 AI 回答问题的文档使用流程

```
用户提问
    ↓
1. 判断问题类型
   - 架构问题 → 查阅 architecture.md
   - 组件使用 → 查阅 components.md
   - API 调用 → 查阅 api.md

   - 更新日期

4. 选择引用文档

   - 避免引用归档文档

5. 在回答中说明
   - "根据 [文档名]（状态：已实施 ✅）..."

- 验证链接有效性

**每月**:

**每季度**:
- 重构文档结构（如需要）
- 收集开发者反馈
- 优化文档导航

### 7.2 文档驱动开发

**推荐流程**:
```
1. 编写/更新文档 (docs/*.md)
2. 根据文档编写代码
3. 更新代码注释 (JSDoc/Rust doc)
4. 更新 AGENTS.md 摘要
5. 提交代码 + 文档
```

**优势**:
- 文档与代码同步
- 减少事后补文档的工作量
- 提高代码质量（文档先行，思路更清晰）

### 7.3 示例代码管理

**推荐做法**:
```typescript
// ✅ 推荐：示例代码单独成文件，可测试
// examples/config-usage.ts
import { config } from '@/config';
const theme = await config.get('theme.mode');

// 在文档中引用
// 见 [`examples/config-usage.ts`](examples/config-usage.ts)
```

**避免做法**:
```typescript
// ❌ 避免：示例代码无法测试
// 直接从代码中复制片段到文档
```

## 九、常见问题 FAQ

### Q1: 文档应该多详细？

**A**: 遵循"最小必要"原则：
- AGENTS.md：只包含快速查阅的内容
- docs/*.md：包含完整的使用指南
- 代码注释：解释"为什么"，不是"做什么"

### Q2: 如何保持文档与代码同步？

**A**: 
1. 代码审查时必须检查文档
2. 使用 TypeDoc/rustdoc 自动生成部分文档
3. 定期（每月）审查文档

### Q3: 文档结构何时重构？

**A**: 
- 当文档超过 1000 行时考虑拆分
- 当开发者反馈"找不到信息"时优化导航
- 当重复内容超过 20% 时重组结构
