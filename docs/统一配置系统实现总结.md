## 📋 实现概述

本次重构实现了项目的**统一配置入口**，采用方案 A（增强 ConfigManager），解决了配置系统分散、缺少统一入口的问题。

### 核心改进

1. ✅ **统一配置入口**：创建 `src/config/index.ts` 作为配置系统的单一入口
2. ✅ **类型安全**：完整的 TypeScript 类型定义支持
3. ✅ **配置就绪机制**：提供 `whenReady()` 确保配置加载完成后再访问
4. ✅ **事件订阅**：支持配置变更的发布订阅模式
5. ✅ **向后兼容**：保持与现有 API 的兼容性

## 📁 新增文件

### 1. `src/config/index.ts` - 统一配置管理器

**核心功能**：
- 配置初始化（`initialize()`）
- 配置就绪等待（`whenReady()`）
- 类型安全的配置访问（`getConfigValue()`, `setConfigValue()`）
- 实例配置管理（`getInstanceConfig()`, `updateInstanceConfig()`）
- 配置事件订阅（`on()`, `off()`）

**关键代码**：
```typescript
class UnifiedConfigManager {
  async initialize(): Promise<void>;
  async whenReady(): Promise<void>;
  isReady(): boolean;
  getConfigValue<T extends ConfigKey>(key: T): ConfigValue<T> | undefined;
  async setConfigValue<T extends ConfigKey>(key: T, value: T): Promise<void>;
  on<T extends ConfigEvent>(event: T, callback: Callback<T>): () => void;
}
```

### 2. `src/config/types.ts` - 配置类型定义

**核心类型**：
- `ConfigKey` - 配置键类型（字符串字面量联合类型）
- `ConfigValue<T>` - 配置值类型（条件类型推断）
- `ConfigEvent` - 配置事件类型（`ready`, `change`, `error`）
- `ThemeMode`, `AccentColor`, `Language` - 配置值类型

**示例**：
```typescript
type ConfigKey = 
  | 'preferences.theme'
  | 'preferences.accent_color'
  | 'download.download_path'
  | 'instance_configs';

type ConfigValue<T extends ConfigKey> = 
  T extends 'preferences.theme' ? ThemeMode :
  T extends 'download.download_path' ? string :
  ...;
```

## 🎯 配置系统架构

### 三层存储架构

```
┌─────────────────────────────────────────────────────────┐
│ L1: localStorage (UI 配置 - 立即同步)                    │
│ - 侧边栏宽度、折叠状态                                  │
│ - 主题模式、强调色（通过 zustand persist）              │
│ - 实例视图模式、下载面板状态                            │
└─────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────┐
│ L2: 配置文件 (业务配置 - 防抖异步保存)                  │
│ - 位置：{base_dir}/.smcl/app_config.json                │
│ - 用户偏好：theme, language, enable_animation           │
│ - 下载配置：download_path, concurrent_limit             │
│ - 实例配置：instance_configs.<id>.*                     │
│ - 窗口位置：window_position                             │
└─────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────┐
│ L3: 加密存储 (敏感数据 - 立即异步保存)                  │
│ - 账户 tokens（access_token, refresh_token）            │
│ - 其他敏感凭证                                          │
└─────────────────────────────────────────────────────────┘
```

### 配置初始化流程

```
应用启动
  ↓
invokeAccInit() - 账户系统初始化
  ↓
config.initialize() - 统一配置系统初始化
  ├─ 加载配置文件 (app_config.json)
  ├─ 更新 Zustand Store
  └─ 触发 'ready' 事件
  ↓
Promise.all([...]) - 并行初始化其他 store
  ├─ useThemeStore.init()
  ├─ useInstanceStore.init()
  └─ useDownloadStore.init()
  ↓
应用初始化完成
```

### 配置变更流程

```
用户修改配置
  ↓
config.setConfigValue(key, value)
  ├─ 调用 Rust API (set_config_value)
  ├─ 写入配置文件 (防抖 500ms)
  ├─ 刷新 Zustand Store
  └─ 触发 'change' 事件
  ↓
订阅者收到通知
```

## 📊 配置访问方式对比

| 方式 | 适用场景 | 优点 | 示例 |
|------|----------|------|------|
| **config 单例** | 所有场景，尤其是非 React 环境 | 类型安全、功能完整 | `config.getConfigValue('key')` |
| **Hooks** | React 组件内部 | 响应式更新、使用简单 | `useConfig()`, `usePreferences()` |
| **Zustand Store** | 高级用法 | 直接操作 store | `useConfigStore(s => s.config)` |

**推荐使用**：
- ✅ **首选**：`config` 单例（类型安全、功能完整）
- ✅ **组件中**：Hooks（响应式、简洁）
- ⚠️ **高级场景**：Zustand Store（需要理解内部机制）

## 🎯 解决的问题

### 问题 1：配置访问分散

**之前**：
```typescript
// 多种访问方式混用
const config1 = useConfigStore.getState().config;
const config2 = configManager.getConfig();
const config3 = useConfig().config;
```

**现在**：
```typescript
// 统一使用 config 单例
const config = config.getConfigValue('key');
```

### 问题 2：缺少配置就绪信号

**之前**：
```typescript
// 无法确定配置是否加载完成
const theme = config?.preferences.theme; // 可能为 undefined
```

**现在**：
```typescript
// 等待配置就绪
await config.whenReady();
const theme = config.getConfigValue('preferences.theme');
```

### 问题 3：配置依赖管理

**之前**：
```typescript
// 各 store 独立初始化，无法等待配置
useInstanceStore.getState().init(); // 可能使用默认路径配置
```

**现在**：
```typescript
// 等待配置就绪后再初始化
await config.initialize();
await useInstanceStore.getState().init(); // 使用正确的路径配置
```

### 问题 4：配置变更通知

**之前**：
```typescript
// 无法订阅配置变更
// 只能手动刷新
```

**现在**：
```typescript
// 订阅配置变更
config.on('change', (key, value) => {
  handleConfigChange(key, value);
});
```

## 📚 使用规范

### ✅ 推荐做法

1. **使用统一入口访问配置**
```typescript
const theme = config.getConfigValue('preferences.theme');
```

2. **等待配置就绪后再访问**
```typescript
await config.whenReady();
const value = config.getConfigValue('key');
```

3. **使用类型安全的配置键**
```typescript
config.getConfigValue('preferences.theme');
```

4. **订阅配置变更时清理**
```typescript
useEffect(() => {
  const unsubscribe = config.on('change', handleConfigChange);
  return unsubscribe;
}, []);
```

### ❌ 禁止做法

1. **禁止直接修改配置文件**
```typescript
// ❌ 禁止
fs.writeFileSync(configPath, JSON.stringify(config));

// ✅ 推荐
await config.setConfigValue('key', value);
```

2. **禁止在配置未就绪时访问**
```typescript
// ❌ 禁止
const theme = config.getConfigValue('preferences.theme');
if (theme === 'dark') { ... }

// ✅ 推荐
await config.whenReady();
const theme = config.getConfigValue('preferences.theme') || 'dark';
```

3. **禁止混用多种配置访问方式**
```typescript
// ❌ 禁止
const theme1 = config.getConfigValue('preferences.theme');
const theme2 = localStorage.getItem('preferences.theme');

// ✅ 推荐
const theme = config.getConfigValue('preferences.theme');
```
